1 # C Runtime Library
2 
3 ## Description
4 
5 Modern compilers provide C runtime libraries designed to aid the compiler in
6 performing certain tasks (for GCC, this is the ctrstuff.c in libgcc).
7 Three different functions that most of these libraries perform:
8 - Global Construction (executing functions defined in the ".ctors" section)
9 - Global Destruction (executing functions defined in the ".dtors" section)
10 - Register Exception Framework Section (".eh_frame" section)
11 
12 Although it is possible for the ELF loader to do these tasks for an executable,
13 from a security / containment perspective, it's better for the executable
14 itself to perform these task's in it's own isolated environment. Bareflank,
15 like everything else, needs to perform these same tasks, and thus performs
16 them in the context of the VMM, but can also use this code within a VM if
17 needed. For Bareflank, all of the C++ objects that are globally defined, have
18 constructors / destructors that need to be executed from a global perspective.
19 Each ".eh_frame" section in the ELF binary also needs to be registered with
20 the unwind code to provide exception support.
21 
22 ## How It's Used
23 
24 Each cross compiled shared library is compiled using the bareflank-gcc-wrapper.
25 
26 [bareflank-gcc-wrapper](https://raw.githubusercontent.com/Bareflank/hypervisor/master/tools/scripts/bareflank-gcc-wrapper)
27 
28 If you look at a GCC compiler (and even Clang/LLVM), both compilation
29 and linking is done with GCC. When linking is required, GCC calls LD (the
30 linker) on your behalf, and adds ctrbegin.o and crtend.o to your
31 executable (which it gets from ctrstuff.c). These additional object files
32 provide the above described functionality. There are 4 different ELF sections
33 that need to be executed here (two of which we currently support)
34 - ctors
35 - dtors
36 - init_array
37 - fini_array
38 
39 GCC appears to use CTORS/DTORS while Clang/LLVM appears to use init_array
40 /fini_array. These sections are all the same; a list of void (*func)(void)
41 function pointers. When each module is loaded, the CTORS/init_array functions
42 all need to be executed, while during destruction, DTORS/fini_array functions
43 need to be executed. Finally, each ELF module has a ".eh_frame" section (even
44 for C code sometimes), that contains stack unwinding information that is
45 needed by the unwind library. The ELF loader gathers the location and size
46 of each of these sections, which in turn is used by the CRT library to
47 setup/teardown a module:
48 
49 [ELF loader](https://github.com/Bareflank/hypervisor/blob/master/bfelf_loader/src/bfelf_loader.c#L1039)
50 
51 To make all of this work, Bareflank's wrapper script acts like
52 GCC and Clang. When "-c" is provided, the code is compiled, and when
53 no "-c" is provided, the code is linked.
54 
55 The create-cross-compiler.sh script compiles this C runtime library as both a
56 static library and a dynamic library (the dynamic version is not used), and
57 places the library in the "sysroot" which is located in
58 ~/opt/cross/x86_64-elf/. When each shared library is compiled, the
59 gcc wrapper script adds the static C runtime library during linking, which
60 means that every cross compiled shared library has the following two functions
61 added:
62 - [local_init](https://github.com/Bareflank/hypervisor/blob/master/bfcrt/src/crt.cpp#L26)
63 - [local_fini](https://github.com/Bareflank/hypervisor/blob/master/bfcrt/src/crt.cpp#L44)
64 
65 Here is an example of the "entry" module for the VMM that has these symbols
66 added by the wrapper script:
67

69 
70 As you can see, these symbols are marked as global, meaning they have relocation
71 entries, and their addresses can be looked up using the ELF loader. The
72 bareflank drivers (also referred to as the driver entry points, located in
73 the bfdrivers folder), is responsible for loading each cross compiled module
74 from the bfvmm folder into memory using the ELF loader. Once everything is
75 loaded / relocated, the driver then locates the local_init function for each
76 module and executes it.
77 
78 [common.c](https://raw.githubusercontent.com/Bareflank/hypervisor/master/bfdrivers/src/common.c)
79 
80 This causes the global constructors to get executed, and registers the
81 ".eh_frame" for the module (to support exception handling). When the driver
82 is "unloading" the VMM, the local_fini function is in turn executed, calling
83 global destructors.
